  • Principles
    • User first

      Placeholder copy - ignore

      Our design language is built as responsive from the ground up. This approach brings with it a certain way of approaching responsive design.

      First, learn the principles on which the EEL has been founded.

      1. Design for users
        We design for the users of the website or application first. We must understand the user's needs well, through use of good research and data to back up design decisions. All interactions must perform quickly with timely feedback in all environments. Buttons must show immediate reaction. Page load times must be minimized as much as possible. On-page actions, javascript, and ajax should be speedy in all environments.
      2. Modularity
        Modularity should be at the core of EdWeb to enable re-use of the design system, patterns and, ultimately, the code.
      3. Technical independence
        EEL should be technically agnostic built using web standards principles.
      4. Build the interface as the brand
        Through intelligent and creative use of EEL's brand typography and colour usage, build the brand into the very heart of EdWeb. Brand aesthetics should be obvious in all environments.
      5. Design for content
        Design for content, not devices or screen widths. Whilst employing responsive design principles, we must build to content first, and devices second. We should aim for content parity across media and devices.
      6. Simplicity and clarity at the core
        Wherever possible, we must try to reduce complexity. This includes functionality, user experience and, above all, copywriting. Labelling should be clear and building upon Principle 1.
      7. Lower the on-ramp
        EEL should promote rapid design and development by providing a tool-kit of patterns for site commissioners, content creators, designers and developers.
      8. Build sustainable components
        The design patterns should be agnostic to container width allowing placement across the design system allowing for future-friendly adoption by a myriad of screen sizes.
      9. Place the Edinburgh Experience at the heart of new site commissioning and development processes. The defacto standard and starting point for all new site creation. Brand aesthetics should be obvious in all environments.
      10. Easy to learn and pleasant to use.
        EEL should not be difficult to use and be where site owners and developers need it most: commissioning, planning, designing or developing. EEL should easily provide answers to design problems.
      11. Smart defaults
        Where there are transactions required by the user (search, filling and submitting forms, for example), we ensure the default settings are devoid of ambiguity.
    • Design for content

      Design for content, not devices or screen widths. Whilst employing responsive design principles, we must build to content first, and devices second. We should aim for content parity across media and devices.

    • Consistency of user experience

      Wherever possible we should use the same language and the same design patterns — this helps people get familiar with our services. But, when this isn’t possible, we should make sure our underlying approach is consistent. So our users will have a reasonable chance of guessing what they’re supposed to do.

      This isn’t a straitjacket or a rule book. We can’t build great services by rote. We can’t imagine every scenario and write rules for it. Every circumstance is different and should be addressed on its own terms. What unites things, therefore, should be a consistent approach — one that users will hopefully come to understand and trust — even as we move into new digital spaces.

    • Standards
    • Modular, extensible and open

      We should share what we’re doing whenever we can. With colleagues, with users, with the world. Share code, share designs, share ideas, share intentions, share failures. The more eyes there are on a service the better it gets — howlers get spotted, better alternatives get pointed out, the bar gets raised.

      Partly because much of what we’re doing is only possible because of open source code and the generosity of the web design community. So we should pay that back. But mostly because more openness makes for better services — better understood and better scrutinised. If we give away our code, we’ll be repaid in better code. That’s why we’re giving away all this...

  • Foundations
    • Bootstrap

      Bootstrap is the most popular HTML, CSS, and JS framework for developing responsive, mobile first projects on the web. Bootstrap makes front-end web development faster and easier. It's made for folks of all skill levels, devices of all shapes, and projects of all sizes.

      We built the EEL on top of Bootstrap so you have a solid foundation to build from. Using Bootstrap also gives you access to a huge community of developers and content creators to help you when you need it.

      Bootstrap

    • Responsive layout

      Responsive design is generally the best way to deliver a consistent experience across the wide (and ever-increasing) range of devices that your customers are using to access your services. EdWeb is built in a responsive, mobile-first way. Bootstrap's grid system allows you to quickly and easily add new components to your templates that will work well on almost any device.

    • CSS pre-processor

      Bootstrap ships with vanilla CSS, but its source code utilizes the two most popular CSS preprocessors, Less and Sass.

      The EdWeb implementation uses Less.

    • Contacts

      Status: Proposed

      Code: Bootstrap

      Colour themed: no

      A structured content block that groups an individual's or functional area's contact details. Uses standard Bootstrap elememts.

      Components

        Personal contact asset has the following attributes (+ schema.org reference):
      1. First name (mandatory; maps to givenName)
      2. Additional name (maps to additionalName)
      3. Family name (mandatory; maps to familyName)
      4. Title (maps to honorificPrefix)
      5. Role (maps to jobTitle)
      6. Awards (maps to award)
      7. Image (Be able to associate an image with a contact asset; maps to image)
      8. Address (maps to address.streetAddress)
      9. Locality (maps to address.addressLocality)
      10. Postcode (maps to address.postalCode)
      11. Organisation details (title for following subgroup)
        1. Organisation 1
        2. Organisation 2
        3. Organisation 3
      12. Work phone
      13. Fax
      14. Email
      15. Contact times
      16. Extra details - These extra details are categorised, and the nature of the fields to complete depends upon the category selected (in each case the category maps to ContactPoint.Type):
        1. Additional work phone (fields: Label (maps to ContactPoint.alternateName), Number (maps to ContactPoint.telephone)) - Both mandatory fields
        2. Mobile (field: Number (maps to ContactPoint.telephone)) - Mandatory field
        3. Social media (fields: Link text (maps to ContactPoint.alternateName), URL (maps to ContactPoint.sameAs)) -
        4. Web (fields: Link text (maps to ContactPoint.alternateName), Link title (maps to ContactPoint.description), URL (maps to ContactPoint.sameAs)) - where the URL provided is generating a link using the link text on output, including a title attribute. Link text and URL mandatory

    • Video

      Functionality

      • Video at larger sizes should play in place.
      • Video at lower sizes should be a placeholder that launches an overlay player. See Modal module.

      Accessibility considerations

      • Text transcripts should be provided for videos.
      • All videos should be captioned.
      • Video should never automatically play.
      • Alternative content such as a title or description should be provided to all assistive technology to handle video if it can’t be played.

  • Editing this documentation
    • Editing src.htm

      All content for this documentation is stored in a single HTML file, src.htm

      src.htm is arranged as a nested list, the hierarchy of which is what gives the documentation its structure.

      Each item or "page" is comprised of up to five parts:

      • .title - This is the page title
      • .desc - This is the description
      • .demo - This describes what shows in the three previews at the bottom. It will take either HTML code or a URL
      • .snippet - This is code preview on the right. (wrap HTML in <PRE> tags to have it display correctly *). If left empty the snippet will take its HTML from the .demo
      • .textpage - This is an optional div for creating a full screen page. If it exists, description, snippet and previews will not be displayed. This field accepts any HTML.

      * snippets will need preformatted HTML, like so:

      <pre>
      
  &lt;li&gt; &lt;/li&gt;
      
</pre>

      An example of a section with a single page

      <li><div class="sectionTitle">** SECTION TITLE **</div>
      
  <ul>
      
    <li>
      
      <div class="title"> ** PAGE TITLE (text) ** </div>
      
      <div class="desc"> ** PAGE DESCRIPTION (html) ** </div>
      
      <div class="snippet"> ** SNIPPET (optional) (preformatted html) ** </div>
      
      <div class="demo"> ** DEMO (html or url) ** </div>
      
      <div class="textpage"> ** TEXT PAGE CONTENT (optional) (html) ** </div>
    </li>
      
  </ul>
      
</li>
